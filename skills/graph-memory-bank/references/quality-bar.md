# Quality Bar (для графовой памяти)

## Что считать “хорошо”

- Узлы атомарные: один файл = одна мысль/сущность/контракт.
- Текст “про дело”: без воды, но с якорями (paths, имена классов/таблиц/экранов).
- Граф двунаправленный:
  - из БП видно, какие экраны/процедуры/таблицы участвуют
  - из экрана видно, в каких БП он используется
- Frontmatter заполнен стабильно, без “плавающих” схем.

## Проверка узла (чек-лист)

1. Есть `id/type/title/description/status/tags`.
2. Есть 1-3 inbound ссылки (backlinks) на индексы/хабы.
3. Есть outbound ссылки на соседние сущности.
4. Есть “evidence”: `source_paths` (и/или коммиты/дифф).
5. Если замечен риск/неопределенность: это явно помечено отдельной секцией `Риски/заметки`.

## Как писать, чтобы агенту было дешево по токенам

- Ставь “точки входа” сверху: список путей/классов/таблиц.
- Избегай длинных рассказов; лучше 5 коротких буллетов.
- Выноси списки/справочники в отдельные узлы и ссылками связывай обратно.
- Не пытайся описать весь домен в одном месте: делай хабы и подузлы.

## Красные флаги (что нужно чинить сразу)

- Дубликаты `id` (ломает ссылочную целостность для агентов).
- Узел без ссылок (orphan), если он не “временная заметка”.
- Узел на 800+ строк без явных причин.
- “Generated” секция отредактирована вручную.
